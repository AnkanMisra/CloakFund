# CloakFund Implementation Log & Guide

This document serves as a detailed ledger of the development process for CloakFund. It explains **what** was built in each phase, **how** it was implemented technically, and **why** specific design choices were made.

---

## Phase 0: Preparation & Access

### What was done
- Created a dedicated development branch (`feature/phases-implementation`).
- Generated `.env.example` to track required environment variables.
- Created `docs/SECRETS_SETUP.md` to establish guidelines for managing API keys and handling API rate limits/unavailability using mock endpoints.
- Created `docs/test_wallets.md` containing mock Base Sepolia testnet addresses for the Sender, Receiver (ENS identity), and Treasury Vault.

### How it was done
- Scaffolded standard markdown and environment template files in the root and `docs/` directories.
- Defined explicit fallback flags in `.env.example` (e.g., `USE_MOCK_BITGO=true`) to toggle between real services and mock data.

### Why it was done
- **Security:** Ensures developers don't accidentally commit real API keys or private keys to version control.
- **Reliability:** By defining a mocking strategy upfront, the team can continue developing or demoing the app even if a sponsor's testnet API goes down during the hackathon.
- **Testing:** Pre-defining test wallets creates a standardized environment for testing the entire flow without draining real funds.

## Phase 1: Core Rust Backend (Stealth Generator Module)

### What was done
- Initialized the Rust backend workspace (`cargo new rust-backend`).
- Added necessary cryptographic dependencies (`k256`, `sha3`, `hex`, `rand`, `elliptic-curve`).
- Implemented the core stealth address generation logic in `rust-backend/src/stealth.rs`.
- Wrote unit tests to validate the format and correctness of the generated Ethereum addresses and ephemeral public keys.
- Documented the expected inputs and outputs in `docs/CRYPTO_TEST_VECTORS.md`.

### How it was done
The stealth generation uses Elliptic Curve Diffie-Hellman (ECDH) on the `secp256k1` curve:
1. **Input:** The recipient's public key (derived from their ENS or wallet).
2. **Ephemeral Key:** The sender generates a random ephemeral private/public keypair.
3. **Shared Secret:** The sender performs ECDH by multiplying their ephemeral private key with the recipient's public key.
4. **Key Derivation:** The shared secret is hashed using `Keccak256` to derive a deterministic scalar.
5. **Stealth Key:** This scalar is multiplied by the Generator point (G) and added to the recipient's original public key to form the new **Stealth Public Key**.
6. **EVM Address:** The final stealth Ethereum address is generated by taking the last 20 bytes of the `Keccak256` hash of the uncompressed Stealth Public Key.

### Why it was done
- **Privacy (Unlinkability):** Because the sender uses a random ephemeral key for every transaction, a new, completely unique EVM address is generated every time. On-chain observers cannot link this new address to the recipient's main identity.
- **Self-Custody:** Since the recipient owns the original private key, only they can compute the corresponding private key for this new stealth address (by performing the same ECDH process using the published ephemeral public key).
- **Rust for Crypto:** Rust was chosen for this backend module due to its memory safety and highly audited cryptographic libraries (`k256`), minimizing the risk of side-channel attacks or key leakage.
